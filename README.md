## Overview
Lilo 是一个面向大学入学阶段学生的短信式聊天机器人，目标是帮助学生完成关键入学流程，并降低 **summer melt** 风险——即学生虽然已经被大学录取，但最终未能顺利入学。

对我来说，这个项目最有价值的地方，不只是“聊天机器人”这个形式本身，而是它所代表的一个真实产品命题：

**当用户处在高压力、信息不对称、任务复杂的现实场景中，AI 如何以可信、清晰、可执行的方式帮助他们完成目标？**

这也是我最关注的 AI 产品方向之一：  
不是为了展示模型能力而做 AI，而是让 AI 真正进入真实世界、支持真实用户、解决真实问题。

---

## Why This Project Matters
大学录取并不意味着学生一定会顺利入学。对于许多学生来说，录取之后仍然要面对一系列复杂而关键的任务，例如：

- 阅读并理解学校要求
- 处理材料、表格和截止日期
- 判断自己下一步该做什么
- 在焦虑、拖延或信息混乱中继续推进流程
- 在需要帮助时找到正确的支持渠道

传统的信息载体，例如网页、邮件或 FAQ，往往默认用户已经知道自己在找什么，也默认用户有足够耐心去筛选信息。但现实中的用户并不是这样运作的。

Lilo 的价值在于，它试图把“信息获取”变成“对话支持”——  
让系统不仅回答问题，还能陪伴用户理解任务、降低门槛、推动下一步行动。

## The Core Product Problem
这个项目最核心的问题，不是“如何让机器人回答问题”，而是：

**如何让一个对话式系统在用户最容易迷失、拖延、焦虑或不知所措的时候，提供正确、可信且可执行的帮助。**

这意味着系统必须同时处理三层问题：

### 信息层
用户需要知道：
- 截止日期是什么
- 要交什么材料
- 流程下一步是什么
- 某个政策或要求意味着什么

### 任务层
用户需要知道：
- 我现在最应该做什么
- 哪个任务最优先
- 我是否已经完成了关键步骤
- 如果我卡住了，接下来该怎么办

### 情绪层
用户可能还会经历：
- 焦虑
- 拖延
- 不确定
- “我是不是搞错了”的担心
- 不知道该向谁求助的无助感

真正好的对话式 AI 产品，必须同时对这三层有感知，而不是只停留在 FAQ 式问答。

---

## Understanding the Users
Lilo 所面向的用户，并不是理想化、冷静、流程清晰的使用者，而是真实处于教育过渡阶段的学生。

他们可能会出现以下情况：

- 对大学流程并不熟悉
- 对截止日期和表格要求感到压力
- 并不知道自己缺失的是哪一部分信息
- 没有耐心阅读冗长网页内容
- 即使知道信息，也未必能转化为行动
- 遇到复杂问题时，不知道该联系谁

这对产品设计提出了一个很典型的 HCI 问题：

**系统不仅要“信息正确”，还要“交互合适”。**

因为用户的问题往往并不会以结构化方式表达，他们更可能说：

- “我现在该做什么？”
- “这个是不是过期了？”
- “我不知道我还差什么。”
- “我有点慌。”
- “我找不到那个表格。”

这些表达背后并不只是一个问题，而往往混合了：
- 意图不明确
- 任务状态不清楚
- 情绪压力较高
- 下一步行动模糊

---

## Product Goals
从 AI 产品与用户体验角度看，我认为 Lilo 这类系统的核心目标包括：

- 帮助学生理解当前阶段的关键信息
- 降低流程理解和执行的门槛
- 用更自然的交互方式替代高负担的信息查找
- 将“问答”升级为“任务推进”
- 在必要时引导用户寻求进一步支持
- 通过更及时、更贴近用户的沟通降低 summer melt 风险

---

## Key UX Challenges
这个项目的难点，不在于让系统“会说话”，而在于让它在真实环境中有用。

### 1. Users rarely ask perfect questions
用户不会总是用标准表达提问。  
很多时候，他们的问题是零散的、模糊的、上下文缺失的，甚至带有情绪色彩。

### 2. Answering is not enough
即使系统回答了问题，也不代表用户就能继续推进流程。  
产品必须思考：这条回复有没有真正帮助用户进入下一步？

### 3. Trust is fragile
教育场景中的错误信息成本很高。  
一旦用户觉得回复不准确、模棱两可或太“像机器”，信任就会迅速下降。

### 4. Not every case should be automated
有些问题适合自动化处理，有些问题则需要转接到真人支持。  
因此系统不能只追求自动回复率，还要考虑边界判断和转接机制。

---

## From Chatbot to AI Agent
我认为 Lilo 更值得被理解为一个 **task-oriented conversational agent**，而不只是传统意义上的 chatbot。

因为它的价值不只在于“回答”，而在于根据用户处境做出更合适的下一步支持，例如：

- 澄清用户问题
- 判断用户可能处于哪个流程阶段
- 提供与当前状态相关的说明
- 给出清晰的下一步建议
- 必要时转接人工支持

这也是我理解的 Agent 体验设计重点之一：

**Agent 的核心不是“像人说话”，而是围绕用户目标采取合适行动。**

---

## Relationship to LLMs
从技术框架来看，Lilo 这类系统与大语言模型高度相关，因为用户输入天然是自然语言，而且往往不规范、不完整、上下文依赖强。

LLM 在这里能提供的关键能力包括：

- 自然语言理解
- 多样化表达的意图识别
- 在上下文中生成更自然的回复
- 将复杂信息转化为更易理解的话语
- 支持多轮交互中的连续性

但与此同时，我也认为 LLM 在这类产品中不能被“神化”。  
在高风险教育场景中，模型能力必须和产品策略一起设计，否则很容易出现：

- 幻觉
- 过度自信
- 给出看似自然但并不准确的建议
- 让用户误以为自己已经完成关键步骤

所以，AI 产品设计的关键从来不只是“接入 LLM”，而是设计一整套可信的交互与信息机制。

---

## Relationship to RAG
如果将 Lilo 与学校的真实知识源连接起来，例如：

- 入学 FAQ
- 学校政策
- 截止日期
- 流程说明
- 表格与资源页面

那么它就非常适合通过 **RAG（Retrieval-Augmented Generation）** 来优化系统表现。

RAG 在这个场景中的价值很明确：

- 提高回答准确性
- 让回复更贴近学校真实信息
- 降低幻觉风险
- 增强用户对系统的信任
- 让模型生成建立在可追溯内容之上

这也是为什么我认为，教育类对话系统很适合作为 **AI + knowledge retrieval + task support** 的典型应用场景。

---

## Human-in-the-Loop Thinking
我特别认同的一点是：  
在教育支持这类真实场景中，好的 AI 产品不应该假装自己能解决一切。

一个成熟的系统应该知道：
- 什么问题可以自动回答
- 什么情况需要进一步澄清
- 什么时候应该提醒用户联系学校工作人员
- 什么场景下不应该继续“硬答”

所以从产品设计上看，**转接机制、边界判断和人工支持接口** 也是体验的一部分，而不是系统失败的表现。

---

## Interaction Design Reflection
从 HCI 角度来看，Lilo 的交互价值主要体现在以下几点：

### 1. Low-friction communication
相比复杂网页导航，对话式入口更轻量，也更贴近日常沟通习惯。

### 2. Contextual support
用户不需要先理解整个信息架构，只需要表达当前困惑，系统再逐步展开帮助。

### 3. Guided progression
系统的理想状态不是一次性倾倒信息，而是根据用户状态一步步引导。

### 4. Emotional accessibility
在高压力场景中，“可接近感”本身就是体验质量的一部分。  
一个语气清晰、结构明确、不过度机械的系统，更容易被用户接受。

---

## Task Flow Perspective
如果从任务流角度理解，这个产品支持的并不是一个单一功能，而是一个连续的用户 journey。  
它所帮助的关键动作可能包括：

- 确认当前阶段
- 获取任务解释
- 判断优先事项
- 解决流程中的具体疑问
- 获得下一步建议
- 在必要时寻求人工帮助

这让我更加确信：

**设计对话式 AI，不能只做 response design，还要做 task design。**
